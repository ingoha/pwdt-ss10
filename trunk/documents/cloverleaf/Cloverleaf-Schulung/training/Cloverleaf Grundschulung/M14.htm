Network Configurator

Mit dem Network Configurator erfolgt die Konfiguration der Anbindung der Subsysteme sowie das Einrichten des Nachrichtenflusses. Bestehende Konfigurationen können geändert werden.

Die Konfiguration der Anbindung und Einrichtung erfolgt immer aus dem Blickwinkel von CLOVERLEAF®, da die Schnittstellen hier letztendlich gestartet und gestoppt werden.

Die Konfigurationsdatei des Network Configurators ist die NetConfig. Diese ist im Verzeichnis $HCISITEDIR gespeichert.

Folgende Konfigurationen werden mit dem Network Configurator durchgeführt:

·       Konfiguration der Inbound Threads

·       Konfiguration der Outbound Threads

·       Definition der Prozesse

·       Konfiguration der Icons/Threads

·       Konfiguration des Routings der Daten Nachrichten

·       Einbinden der Xlates und ggf. TCL Filter

·       Acknowledgement Management

·       Konfiguration des Routings der Reply Nachrichten

·       Konfiguration der Views

·       Konfiguration der Prozess - Logdateien

·       Konfiguration der SMAT Dateien

Thread Properties / Thread Configuration

Je nachdem welche “Rolle” ein Protcoll Thread übernehmen wird, müssen unterschiedliche Reiter ausgefüllt werden. Ein Thread kann folgende „Rollen“ übernehmen:

·       Inbound

·       Outbound

·       Inbound und Outbound

Als Inboundthread benötigt er neben den allgemeinen Informationen des Reiters Thread, Angaben zur Trx ID und evtl. zum Acknowledgement Management. Diese werden auf dem Reiter Inbound eingestellt. Ferner benötigt er Informationen wohin die empfangen Daten geroutet werden sollen. Hierfür steht der Reiter Route Messages zur Verfügung.

 

In der Rolle Outbound benötigt der Thread neben den allgemeinen Informationen des Reiters Thread ggf. Zusatzinformationen (fortlaufender Dateiname, warten auf ACKs, etc.) die über den Reiter Outbound eingegeben werden. Sollen empfangene Acknowledgments zurück zum Sender geroutet werden, müssen diese Angaben auf dem Reiter Route Replies hinterlegt werden.

 

Soll ein Thread sowohl die Rolle Inbound als auch Outbound abdecken sind alle notwendigen Informationen auf den jeweiligen Reitern zu hinterlegen. Erfüllt ein Thread beide Rollen so ist die Interpretation des Thread Status im Netzwerk Monitor nicht trivial. Deshalb wird i. d. R. von dieser Doppelfunktion eines Threads abgeraten und nur in besonderen Fällen angewandt.

 

Im Folgenden werden einige Konfigurationseingaben angesprochen. Auf Grund der Vielfalt der Konfigurationsmöglichkeiten wird an dieser Stelle auf die originalen Handbücher verwiesen. Die Unterlagen der Grundschulung stellen keinen Ersatz für die originalen Unterlagen dar.

Thread Configuration – Reiter Route Messages

·       Auf dem Reiter Route Messages werden die Routen an Hand der Transaction ID (Event, Ereignis, Trx ID) eingerichtet.

·       Jede Route kennzeichnet sich durch einen eindeutigen Namen aus.

·       Jede Route muss ein oder mehrere Ziele besitzen.

·       Für jedes Ziel sind die Route Details zu konfigurieren.

Mit einem Klick der rechten Maustaste auf die Routen (Messages) wird das Kontextmenü aufgerufen. Mit dem Befehl „New Transaction Id“ wird eine neue Route erzeugt.

Bestehen bereits Routen, ändert sich das Kontextmenü. Über den Befehl „Append Id“ wird eine neue Route erzeugt. Die weiteren Befehle des Kontextmenüs sind selbsterklärend.

 

 

Das Fenster Route Messages unterteilt sich in die Layout Pane, welche die Routen mit ihren Zielen anzeigt und den Bereich der Route Properties.

 

Feldname / Button

Beschreibung

Name

Eingabe der Trx ID. Bei HL7 Nachrichten sind das die Events, wobei das Dach (^) als Unterstrich eingegeben werden muss. Beim SAP HCM Standard sind es die Ereignisse wie z. B. NP12I0 etc.

Button „Set as Static Route“

Wird der Button „Set as Static Route“ aktiviert, wird die Route „_HCI_static_route_“ gesetzt. Das bedeutet, dass alle Nachrichten, unabhängig ihrer Trx Id, an das angegebene Ziel geroutet werden.

Wild Card Route

Alle Nachrichten die dem Muster der Trx Id entsprechen, werden an ein Ziel geroutet (Beispiel: Sollen alle ADT Nachrichten geroutet werden, so wäre die Trx Id ADT*).

Add new detail

Wird dieser Button gewählt, wechselt die Ansicht des rechten Fensters von Route Properties zu Detail Properties. Eine Nachricht kann auf 3 verschiedenen Wegen geroutet werden:

Raw                Die Nachricht wird ohne Xlate durch die Engine geroutet. Das Einbinden von TCL oder Java Prozeduren ist optional möglich (z.B. als Filter).
Die Angabe des Zieles wird erwartet. Eine Mehrfachauswahl von Zielen ist möglich, dient aber nicht der Übersichtlichkeit und wird daher nicht empfohlen.

Xlate               Die Nachricht wird unter Verwendung einer Xlate durch die Engine geroutet.
Die Angaben des Zieles und der Xlate werden erwartet. Eine Mehrfachauswahl von Zielen ist möglich, dient aber nicht der Übersichtlichkeit und wird daher nicht empfohlen.
Das Einbinden von TCL oder Java Prozeduren als Pre- oder Post Procs ist optional möglich (z.B. als Filter).

Generate        Das Routing erfolgt ausschließlich unter Verwendung einer TCL oder Java Procedure. Diese Prozedure ermittelt die Route und das Ziel (sehr selten in der Praxis).

Thread Configuration – Reiter Route Replies

 

Der Reiter Replies ist analog dem Reiter Route Message aufgebaut. Lediglich die Funktion Generate Route Details steht hier nicht zur Verfügung.

 

CLOVERLEAF® routet die ACKs ausschließlich an den Ursprungsthread zurück.

 

Bevor ACKs an das sendende System zurück geroutet werden, sollte der Sinn der ACK Übermittlung durchdacht werden. Handelt es sich lediglich um ein ACK das den korrekten Empfang einer Nachricht bestätigt (Standard), dann wird das ACK vom sendenden System nicht unbedingt benötigt. Auch ein ACK, das die vollständige Verarbeitung eines Datensatzes mitteilt, ist nicht unbedingt vom Sender erwünscht.

 

Beispiel:        Die Aufnahme eines Patienten wird vom KIS gesendet und an die Telefonanlage geroutet. Diese Antwortet mit einem ACK und sendet gleichzeitig die Telefonnummer zurück. In diesem Fall ist es sinnvoll das ACK an das KIS zurück zu übermitteln.

Im rechten Bereich des Fensters können die Prozess Eigenschaften des (links) gewählten Prozesses konfiguriert werden.

 

 

 

Feldname

Beschreibung

EO default

Hinzufügen eines EO Alias zum Prozess (siehe EO Configurator). Diese Einstellung bezieht sich auf alle (Protocol) Threads des Prozesses.

EO Xlate

Hinzufügen eines EO Alias zum Prozess (siehe EO Configurator). Diese Einstellung bezieht sich auf den Xlate Thread des Prozesses.

Automatic Log Cycling

Die Aktivierung dieser Funktion bewirkt, dass das Logfile des Prozesses nach Erreichung der angegebenen Größe im Feld „Threashold in KB“ automatisch gecycled wird. Das bisherige Logfile wird umbenannt und der Name wird mit der Bezeichnung old erweitert. Ein neues Logfile wird automatisch angelegt.

Disk-based Queueing

Aktivierung der Funktion Disk-based Queueing.

Während der normalen Verarbeitung werden die Queues der Threads und Prozesse im Hauptspeicher gehalten und zusätzlich durch die Recovery Datenbank abgesichert. Disk-based Queueing speichert die Daten der Nachricht (nicht die Metadaten) auf die Festplatte um so weniger Hauptspeicher zu benötigen.

Diese Option ist dann empfehlenswert, wenn der Prozess eine sehr hohe Auslastung anzeigt oder wenn die Nachrichten extrem groß sind.

Translation Throttling

Durch diese Funktion kann die Anzahl der Nachrichten begrenzt werden, die pro Aktion vom Xlate Thread verarbeitet werden sollen. Normalerweise verarbeitet der Xlate Thread alle Nachrichten, die zur Verarbeitung anstehen. Sehr viele Nachrichten in der Queue können dazu führen, dass keine Prozessorzeit den anderen (Protocol) Threads mehr zugeteilt werden kann.

Mit den Einstellungen Minimum, Maximum und % kann eine gleichmäßigere Verteilung der Prozessorzeit erreicht werden.

Minimum        Anzahl der Nachrichten die vom Xlate Thread verarbeitet werden dürfen. Übersteigt die Anzahl der zu verarbeitenden Nachrichten diesen Wert, so greift die Angabe des Prozentwertes.

Maximum       Anzahl der Nachrichten die maximal vom Xlate Thread verarbeiten werden dürfen. Ist das Minimum überschritten und die Prozentangabe ermittelt einen Wert oberhalb des Maximums, so wird maximal die hier angegebene Anzahl von Nachrichten verarbeitet.

Percent          Anzahl in Prozent der Nachrichten, die verarbeitet werden sollen sobald das Minimum überschritten wurde.

Beispiel:        Folgende Werte sind gesetzt: Min: 100, Max: 200, % 50.
Eingelesen wird eine Datei mit 660 Nachrichten.

Step 1:           50 % von 660 = 330, Max. 200 --> 200 Nachrichten werden verarbeitet, Rest 460.

Step 2:           50 % von 460 = 230, Max. 200 --> 200 Nachrichten werden verarbeitet, Rest 260.

Step 3:           50 % von 260 = 130, Max. 200 --> 130 Nachrichten werden verarbeitet, Rest 130.

Step 4:           130 liegt zwischen dem Minimum und dem Maximum und werden somit verarbeitet.

Run Translation Procs
in start-mode

Pre- oder Post-Xlate TCL Prozeduren werden im Startup mode ausgeführt sobald ein (Protocol) Thread startet. Diese Funktion ist sinnvoll, wenn in den Prozeduren globale Variablen zur Anwendung kommen.
