Database Administrator

Zur Absicherung der Nachrichten verwendet CLOVERLEAF® den Raima Database Manager. Die Raima Datenbank setzt auf ein Netzwerk orientiertes Datenbank Modell auf. Durch diese Architektur ist die RDM wesentlich schneller und kleiner als eine relationale Datenbank und besitzt zudem einen wesentlich kleineren Overhead.

Die Konfigurationsdateien der RDM werden im Verzeichnis $HCISITEDIR/exec/databases gespeichert.

 

CLOVERLEAF® unterscheidet die Datenbanken:

·       Recovery Datenbank

·       Error Datenbank

Recovery Datenbank

Die Recovery Datenbank wird eingesetzt um Nachrichten, die sich in der Verarbeitung befinden, wieder herzustellen bzw. bereit zu halten. Sollte es in der Verarbeitung zu einer Störungen in Form von

o      Prozessunterbrechung

o      Thread Stopps

o      Fehlende Connection (keine Auslieferungsmöglichkeit)

o      Stromausfall

kommen, so werden die Nachrichten solange in der Recovery Datenbank bereitgehalten, bis die jeweilige Störung behoben ist und die Nachricht erfolgreich ausgeliefert werden konnte.

 

Ø        Jede Nachricht die CLOVERLEAF® empfängt wird in die RDB gespeichert (wenn aktiviert).

Ø        Jede Nachricht bekommt eine eindeutige Message ID.

Ø        Jede Nachricht bekommt einen Verarbeitungsstatus, der exakt die Phase der Verarbeitung widerspiegelt.

Ø        Sobald die Nachricht erfolgreich ausgeliefert wurde, entfernt CLOVERLEAF® diese aus der Recovery Datenbank.

 

Error Datenbank

 

Die Error Datenbank speichert Nachrichten

o      deren Format fehlerhaft ist

o      die in der Translation einen Fehler auslösten

o      die unerwartet eintreffen

o      die nicht auslieferbar sind, da z. B. keine Route eingerichtet wurde.

 

Die Error Nachrichten verbleiben solange in der Error Datenbank bis sie manuell vom Administrator entfernt werden.

 

 

 

Database Selection

Selection

Beschreibung

Database

In der Database Selection wird angegeben, welche Datenbank bearbeitet werden soll. Auswahlmöglichkeit zwischen Recovery und Error Datenbank.

Output

In der Output Selection können folgende Einstellungen zu der gewünschten Ausgabeform eingestellt werden:

Screen

Die Ausgabe soll in einem neuen Fenster erscheinen.

File

Die Ausgabe soll in eine Datei gespeichert werden.

Normal Format

Die Ausgabe enthält lediglich die Basic Informationen. (Nur in Verbindung mit der Ausgabe Screen aktivierbar.)

Long Format

Die Ausgabe enthält zu den Basis Informationen nützliche Details zur Sortierung. (Nur in Verbindung mit der Ausgabe Screen aktivierbar.)

Ultra Long Format

Die Ausgabe enthält sowohl die Metadaten jeder Nachricht, wie auch die Nachricht selber. (Nur in Verbindung mit der Ausgabe Screen aktivierbar.)

Im Ausgabe Fenster wird die Option Word Wrap angeboten. Diese Option aktiviert den Zeilenumbruch.

Binary Length Encodet

Die Ausgabe erfolgt im Format Binary Lenght Encodet, d. h. jeder Nachricht wird ein zehnstelliger Zähler voran gestellt, der die Länge der Nachricht angibt. Binary Length Encodet ist default.

Newline Terminated

Die Ausgabe der Nachrichten erfolgt zeilenweise. Jeder Datensatz wird durch ein Newline Zeichen beendet. Dieses Format sollte nicht verwendet werden, wenn die Nachricht selber Newline Zeichen beinhalten könnte.

Filename + Button Öffnen

Ist die Ausgabeform File gewählt, muss der Speicherplatz und ein Dateiname eingegeben werden. Der Button Öffnen öffnet das Browser Fenster mit dem Dateisystem.

Overwrite existing file

Diese Option bietet die Möglichkeit eine bestehende Datei zu überschreiben. Sollte diese Option nicht aktiviert sein, dann werden die neuen Datensätze an die bestehende Datei angefügt. (Nur in Verbindung mit der Ausgabe File aktivierbar.)

Database order

Der Ausgabe kann eine Sortierreihenfolge mitgegeben werden.

Ascending inbound arrival time        Aufsteigend nach der Inbound Ankunftszeit

Ascending outbound arrival time      Aufsteigend nach der Outbound Ankunftszeit

Ascending recovery time                   Aufsteigend nach der Wiederherstellungszeit

Ascending xlate start time                 Aufsteigend nach der Xlate Startzeit

Map data

Diese Option bietet die Möglichkeit die Daten zu konvertieren. Die gängigste Form ist die Konvertierung von ASCII zu EBCDIC. Sollte diese Option gewählt werden, so erwartet CLOVERLEAF® die Eingabe des Filters im nebenstehenden Feld.

Delete

Abfrage was mit den selektierten Daten geschehen soll.

Delete Send Messages                         Alle Nachrichten in der Ausgabe werden gelöscht

DO NOT Delete Send Messages         Die Nachrichten werden lediglich angezeigt. (DEFAULT)

Attributes

In der Attributes Selection könnten Filter aktiviert werden, um die Auswahl der Nachrichten einzugrenzen.

Destination

Einschränkung der Ausgabe auf den empfangenen Thread.

Source

Einschränkung der Ausgabe auf den sendenden Thread.

Owner

Einschränkung der Ausgabe auf den derzeitigen Eigentümer Thread. (Wird i. d. R. nur vom Support benötigt.)

Message Type

Unterscheidung zwischen Data und Reply Nachricht, i. d. R. sind die Data Nachrichten gemeint.

Message State

Einschränkung auf den Status einer Nachricht. (siehe unten)

Message ID

Einschränkung auf die Message ID. Achtung, während des Durchlaufs durch die Eingine verändert sich die Message ID einer Nachricht.

Operation

Aktivierung des ausführlichen Mode.
