Translation / Übersetzung

Der Translation Configurator übersetzt die eingehenden Datensätze in die geforderten Formate der abgehenden Datensätze. Der Translation Configurator bietet eine Vielzahl von Aktionen an, die vergleichbar sind mit den Möglichkeiten einer Programmiersprache (Variable, Konstanten, Schleifen, IF-Abfragen, mathematische Operationen, etc.). Der Translation Configurator ist einer der mächtigsten Configuratoren des Kommunikationsserver CLOVERLEAF®.

Die Konfigurationsdateien des Translation Configurators werden im Verzeichnis $HCISITEDIR/Xlate gespeichert.

 

Translation Thread (Xlate Thread)

Der Translation Thread nimmt die Nachricht vom Inbound Thread entgegen. Anschließend parsed er die Nachricht, so dass er direkt die Inhalte der Nachricht zur Verfügung hat und übergibt sie in das gewünschte Feld des Ausgangsformat. Der Translation Thread arbeitet ausschließlich mit Engine Messages, d.h. in dieser Phase des Datendurchlaufes existiert kein Bezug zum sendenden oder empfangenden Subsystem. Der Translation Thread empfängt und sendet alle Nachrichten via ICL (siehe CLOVERLEAF® Prozess & Thread Architektur).

Pre TCL / Java

Der xlate-Action-Preprocessor ist ein UPOC (User Point of Control) zum Einbinden einer TCL Prozedure oder einer Java Klasse. Diese wird vor der eigentlichen Übersetzung durchgeführt und bezieht sich nur auf die im Bereich Source angegebenen Felder. Beispiel: Formatieren eines bestimmten Datumsformats oder abschneiden der Jahreszahl.

Post TCL / Java

Der xlate-Action-Prostprocessor ist ein UPOC (User Point of Control) zum Einbinden einer TCL Prozedure oder einer Java Klasse. Diese wird nach der eigentlichen Übersetzung durchgeführt und bezieht sich nur auf die im Bereich Destination angegebenen Felder. Beispiel: Herausschneiden der Jahrhundert Angabe eines Datums.

Queues – Message Flow durch die Queues

 

CLOVERLEAF® verarbeitet alle Nachrichten in der FIFO (First In First Out) Reihenfolge. Alle Nachrichten bekommen die gleiche Priorität. Nachrichten mit der gleichen Priorität werden aus den Queues in der Reihenfolge ausgelesen, wie sie eingetroffen sind, die Älteste zuerst.

 

Q4        Route Detail Queue

Die Hauptaufgabe der Route Detail Queue besteht darin, alle Übersetzungen aus einer Eingangsnachricht solange zwischen zu speichern, bis alle neuen Ausgangsnachrichten erzeugt wurden. Sollte bei einer Übersetzung ein Fehler auftreten, dann kann der Anwender über eine UPoC (User Point of Control) selber entscheiden, wie mit diesem Fehler umzugehen ist. Möglich wäre z. B. das Verwerfen aller neuen Ausgangsnachrichten und speichern der Eingangsnachricht in die Errordatenbank.

 

Q5        Partial Queue

Nachdem alle neuen Ausgangsnachrichten die Partial Queue erreicht haben, wird der Eingangsdatensatz aus der Recovery Datenbank entfernt. Anschließend werden die Nachrichten an die Post-Translation Queue weitergegeben.

Sollte während der Übermittlung ein Systemfehler auftreten, z. B. Stromausfall, kann die Engine ermitteln, bei welchem Vorgang die Störung eingetreten ist. Findet die Engine den Eingangsdatensatz in der Recovery Datenbank wieder, wird sie die Übersetzungen neu durchführen. Findet Sie keinen Eingangsdatensatz, dann wird sie die Übermittlung der Nachrichten von der Partial Queue an die Post-Translation Queue fortsetzten.

 

Q6        Post-Translation Queue

Der Xlate Thread stellt für jeden Outbound Thread eine eigene Post-Translation Queue zur Verfügung. Diese Post-Translation Queue hat die Aufgabe die erstellen Nachrichten an die jeweiligen Outbound Threads zu übergeben. Gewöhnlich ist diese Auslieferung erfolgreich, es sei denn der Outbound Thread ist gestoppt oder der Prozess, der den Outbound Thread beinhaltet, ist nicht gestartet. In diesem Fall verbleiben die Nachrichten in der Post-Translation Queue.

CONTINUE, SEND & SUPPRESS

Ø        Erzeugen von mehreren Ausgangsdatensätzen aus einem Eingangsdatensatz.

Ø        Die Reihenfolge des SEND bzw. CONTINUE Befehls hat erhebliche Auswirkungen auf die Erzeugung der Datensätze.
