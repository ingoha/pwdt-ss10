Network Monitor

Der Network Monitor ist das wichtigste Administrations - Tool des Kommunikationsserver CLOVERLEAF®.

Im Network Monitor werden alle Prozesse und Threads gestartet, gestoppt und überwacht. Störungen im Datenfluss können schnell lokalisiert werden.

 

Der Network Monitor generiert automatisch anhand der Informationen in der Datei NetConfig ein Network Control und Monitoring Panel.

 

View                                          Arbeitsfläche View / List / Grid                                                Prozess

Options - Menü

Menüeintrag

Funktion

Edit

Ruft das Fenster Network Monitor Properties auf. Dieses unterteilt sich in die Reiter:

General               Im Bereich Polling Intervals wird angegeben in welchen Intervallen der Netzwerk Monitor den Status der Threads und des Monitor Daemon aktualisiert.

Im Bereich Alert Settings wird angegeben ob das Fenster Alert, automatisch bei jedem Alarm angezeigt und der neue Alarm ausgewählt sein soll.

Engine Output    Festlegen der Erscheinung, der Aktualisierung und des Umfanges der Dialogbox des Command and Engine Output. Über den Parameter netmon_auto_load_logs=true (client.ini/general) können die Reiter für die Prozesse zugeschaltet werden.

Depth Colors     Mit unterschiedlichen Farben kann die Anzahl der Nachrichten in den Queues der Threads im Netzwerk Monitor angezeigt werden. Hierdurch kann auf einen Blick ein Stau im Datenfluss erkannt werden. Auf diesem Reiter Depth Colors können die Farben frei definiert werden. Diese Einstellung ist Clientbezogen.

 

Prozesse

Standardmäßig wird das Process Monitor Panel entlang der rechten Fensterseite des Netzwerk Monitors angezeigt. Das Process Monitor Panel enthält für jeden Prozess einen eigenen Button. Über diesen Prozess - Button ist die gesamte Steuerung des Prozesses möglich. Das Kontextmenü eines Prozess wird mit der rechten Maustaste geöffnet.

Threads in der View

Die Arbeitsfläche des Network Monitors enthält eine grafische Darstellung der Threads. Über die linke Spalte View lassen sich verschiedene Ansichten definieren. Standardmäßig ist die View „all_threads“ aktiviert. Weitere Views können individuell erstellt werden.

 

Ein Thread besteht aus den Teilen

·       Thread Indikator

Der Thread Indikator bildet den Zustand der Verbindung zum Subsystem ab. Die Darstellungszeichen sind analog den Zeichen einer Audio Wiedergabe (Pause, Stopp, Start, etc.).

·       Thread Name

Der Thread Name identifiziert das Subsystem.

·       Thread Connection Status

Der Thread Status kann folgende Werte annehmen:

 

Status

Bedeutung

Dead

Der Thread ist nicht gestartet.

Initializing

Der Thread wurde gerade gestartet und CLOVERLEAF® bereitet seine Einsatzbereitschaft vor.

Opening

Der Thread versucht eine Verbindung aufzubauen.

Up

Der Thread hat eine Verbindung und ist aktiv.

Timer

Der Thread ist zeitgesteuert, d.h. er läuft nur zu den definierten Zeiten.

Down

Der Thread ist aktiv, baut aber keine Verbindung auf.

Closing

Der Thread wird gerade heruntergefahren.

Error

Die Verbindung lief auf eine Störung aufgrund eines Problems.

INEOF

Eine dateibezogene Connection wurde vollständig ausgelesen.

 

 

Ein Thread kann losgelöst vom Prozess gestartet und gestoppt werden. Bedingung zum Starten des Threads ist jedoch, dass der zugehörige Prozess bereits gestartet wurde, ansonsten wird dieser automatisch mitgestartet. Alle Thread mit aktivierter Option Auto-start connection starten ebenfalls.

Die individuelle Steuerung eines Threads erfolgt über das Kontextmenü des Threads, welches mit der rechten Maustaste aufgerufen werden kann.

Inter-thread Statistics

Sent

Anzahl der Nachrichten, die dieser Thread gesendet hat.

Recvd

Anzahl der Nachrichten, die dieser Thread empfangen hat.

pxqd

Pending Xlate Queue Depth. Anzahl der Nachrichten, die dieser Thread an den Translation (Xlate) Thread gesendet hat und die noch in der Queue sind.

Xlt time

Durchschnittliche Zeit (in Sekunden), die zur Übersetzung einer Nachricht benötigt wird.

T on Q

Time on Queue. Gesamte Wartezeit (Sekunden) in der Queue. Sobald die Nachricht die Queue verlassen hat, wird der Wert wieder zurückgesetzt. Das heißt, dass nur dann ein Wert enthalten ist, solange Nachrichten in den Queues gespeichert sind.

Latency

Gesamtzeit (in Sekunden), die der Thread benötigt hat, um die Nachricht zu verarbeiten.

Tot Lat – T on Q = Processing Time

Thread name

Name des sendenden oder empfangenden Threads.
