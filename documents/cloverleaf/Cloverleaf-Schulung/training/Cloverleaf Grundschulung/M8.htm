HL7 Datensatz

HL7 Version 2.x

HL7 (Health Level 7) ist im Krankenhausbereich in Deutschland und in einigen anderen Ländern zwischenzeitlich ein anerkannter Standard geworden. Der Name HL7 leitet sich aus dem ISO-OSI-Referenzmodel (engl. international standard organisation – open systems interconnection) ab. Das OSI Model beschreibt ein allgemeines abstraktes Modell für die Kommunikation zwischen Datenstationen, d.h. es werden nur die wichtigsten Eigenschaften des Außenverhaltens funktional festgelegt.

Da HL7 auf Applikationsebene die Daten beschreibt, die zwischen den Systemen ausgetauscht werden sollen, wurde der Namen aus diesem Modell abgeleitet.

 

Das HL7 Datensatzformat ist modular aufgebaut. Ausgehend von dem Ereignis (Event) werden Nachrichten (Messages) definiert, die sich aus verschiedenen Segmenten (Segments) zusammensetzen. Die Segmente können optional und oder wiederholbar sein. Das einzelne Segmente setzten sich aus fest definierten Datenfeldern (Fields) zusammen.

 

HL7 ist ein variables Satzformat. Es gibt keine Füllzeichen in einer Nachricht. Die Felder des Segments werden mit einem frei definierbaren Trennzeichen von einander getrennt. In der Vergangenheit hat sich das Pipe-Symbol ( | ) als Trennzeichen im HL7 Standard durchsetzten können. Jedes Feld innerhalb des Segments ist mit den Attributen Typ und Länge belegt. Der HL7 Standard gibt eigene Datentypen vor. Eine Maximumlänge für die Felder ist vorgesehen, wird jedoch im Markt kaum eingehalten. Felder können optional, vorgeschrieben (required) und/oder wiederholbar sein.

Derzeit sind die HL7 Versionen v2.1, v2.2, v2.3, v2.3.1, v2.4 und v2.5 im Markt verbreitet. Von jedem HL7 Standard existieren im Markt eine erhebliche Anzahl von Derivaten. Diese sind auf die unterschiedlichen Interpretationen der Systemlieferanten zurück zu führen, bzw. auf die Adaption der Bedürfnisse des deutschen Gesundheitswesens.

 

Der HL7 Standard bietet die Möglichkeit zusätzliche Felder, Segmente und Nachrichten zu definieren. Hierfür wurden freie Namensräume definiert und reserviert. Zusätzliche Nachrichten und Segmente müssen mit einem Z beginnen (Z-Nachricht und Z-Segmente). Zusätzliche Felder müssen einer Feldnummer >= 90.000 zugeordnet sein.

 

Die HL7 Benutzergruppe Deutschland hat bereits einige Z-Segmente vordefiniert, um die allgemeinen Anforderungen des deutschen Krankenhauswesen abzudecken. Dieses Z-Segment-Register kann auf der Website www.hl7.de eingesehen werden.

 

HL7 Version 3

Die HL7 Version 3 stellt eine neue Generation der HL7 Kommunikation dar. Die Version 3 basiert auf einem Ansatz, der die bisherigen Dogmen der Version 2 komplett verlassen hat. Wurden bei der HL7 Version 2 Nachrichten basierend auf Events definiert, die dann in einem eigenen Format dargestellt wurden, so geht die Grundidee der Version 3 andere Wege. Für die Version 3 wurde eine Prozess Modellierung vorgenommen und das so genannte RIM (Reference Information Model) geschaffen. Es war vorgesehen das aus diesem RIM die einzelnen Nachrichten, je nach Art der Applikation, in XML abgebildet werden sollen. In der Praxis war dieses System jedoch nicht einsetzbar, da durch eine unüberschaubare Anzahl verschiedener Klassen das RIM eine Größe erreichte, die es kaum noch darstellbar und vor allem anwendbar machte.

Aus diesem Grund wurden die verschiedenen Spezialklassen verdichtet und über eine Vererbungshierarchie spezialisiert. Zudem wurde das MDF (Message Definition Framework) Werkzeug geschaffen, um Nachrichten aus der HL7 Version 3 zu extrahieren.

 

Das RIM ist nun auf die 4 Klassenarten Entites (Personen/Einrichtungen), Roles (Rollen), Participations (Mitwirkung) und Acts (Aktion) reduziert. Der Grundgedanke ist, dass immer eine Person/Einrichtung in einer bestimmten Rolle eine Aktion ausführt.

 

Beispiel: Zwei Schwestern nehmen einen Patienten auf und führen ihn in sein Zimmer:

 

In der Version 3 Logik werden jede Schwester als eine Entity in der Rolle "Schwester" definiert. Die Schwestern führen die Aktion “Aufnahme“ gemeinsam für den Patienten (Entity) in der Rolle "Patient" durch. Eine Schwester übernimmt die Participation „administrativen Tätigkeiten“, die andere Schwester übernimmt die Participation „Patient auf sein Zimmer bringen“.

 

 

Durch dieses globale Konzept lassen sich alle Arbeitsabläufe in einem Krankenhaus abbilden.

 

Derzeit gibt es in Deutschland noch keine Schnittstelle die auf HL7 Version 3 basiert. Erste Pilotprojekte mit der Version 3.0 laufen in den Niederlanden. Die dortige Regierung hat einen Auftrag an die niederländische Benutzergruppe zur Definition von Nachrichten für Pharmazie und Infrastruktur (NICTIZ-Projekt) erteilt.

 

Der HL7 Configurator

Der HL7 Configurator bietet die Möglichkeit für alle HL7 Derivate, aufbauend auf dem original HL7 Standard, so genannte HL7 Varianten zu erstellen. Innerhalb einer Variante können die Felder, Segmente und Nachrichten individuell angepasst werden.

 

Die Konfigurationsdateien des HL7 Configurators werden im Verzeichnis $HCISITEDIR/formats/ abgespeichert.
